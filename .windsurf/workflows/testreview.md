# Audit Test Quality

Performs a two-pass audit of test files. Pass 1 flags tests where an LLM likely accepted buggy source code as correct and encoded that behavior as the expected value. Pass 2 identifies material source code sections with no meaningful test coverage. Output is a
structured Markdown report designed to be fed back to an LLM to fix the issues found.

---

## Steps

1. **Identify the test files to audit.**

   If the user specified particular test files or paths, use those. Otherwise, find all test files in the project.

2. **Read all test files and their corresponding source files.**

   For each test file, locate and read the source implementation file(s) it is testing. You need both in context simultaneously for the next step.

3. **Pass 1 — Audit each test file for misleading tests.**

   These tests were largely generated by an LLM that read the source code and wrote tests to match what the code does, without independent knowledge of what it was supposed to do. The primary failure mode is: the LLM observed the code's behavior, assumed it was correct, and encoded that behavior as the expected value. If the code contains a bug, the test passes and permanently masks it.

   For every assertion in every test, ask: "How would a developer who knew the business requirement — but had NOT seen the source code — have written this test?" Flag the test if the answer differs meaningfully from what the test actually asserts.

   If the test code is explicitly commented with a reason that the code could not be properly tested, then you do not need to flag that particular issue.

   **PRIMARY patterns to flag aggressively (A-D):**

   - **A — Expected value mirrors suspicious implementation detail:** The asserted value looks like it was read off the code rather than derived from a spec. Watch for off-by-one results matching an obvious fencepost error; empty collections, zeros, or nulls where a real result is plausible; hardcoded strings matching an implementation literal verbatim where the string looks wrong; dates or counts matching a default/fallback value in the source.

   - **B — Partial result accepted as complete:** The source function does something partially (processes only the first item, skips a transformation step, returns an intermediate state) and the test asserts that partial result as the expected output.

   - **C — Test exercises only code paths that exist, not paths that should exist:** The source is missing a branch (an error case, null guard, conditional) and the test never attempts to reach it, implying the LLM did not notice it was absent.

   - **D — Circular expected value:** The test computes its expected value by calling the same function (or a near-identical expression) being tested. The assertion always passes because both sides move together.

   **SECONDARY patterns (E-H), lower priority:**

   - **E — Tautological assertion:** Always true regardless of implementation. E.g. `expect(true).toBe(true)`, `assert len(result) >= 0`.

   - **F — Mocked-away the thing being tested:** Core logic is stubbed out so the test validates the mock, not the real code.

   - **G — Overly permissive matcher:** `expect.anything()`, `toContain("")`, etc.

   - **H — Exception swallowed:** A try/catch inside the test silently eats failures.

   For each flagged test, produce this block:

   ```
   FILE: <relative path to test file>
   TEST: <describe block > test name>
   PATTERN: <letter and name, e.g. "A — Expected value mirrors suspicious implementation detail">
   SEVERITY: HIGH | MEDIUM | LOW
   FINDING: What the test asserts and why it is suspicious. Quote the relevant assertion
            and the corresponding source line. Be specific.
   ACTION: A single imperative sentence telling the fixing LLM exactly what to change.
           Start with a verb. Reference the specific assertion or expected value to modify.
   ```

   If a file has no issues, output: `FILE: <path> — No issues found.`

   After all files, output:

   ```
   PASS 1 SUMMARY
   Files reviewed: N
   Tests reviewed: N
   Tests flagged: N (PRIMARY: N | SECONDARY: N)
   Dominant pattern: <name>
   ```

4. **Pass 2 — Identify material coverage gaps.**

   Read the source files and identify sections of real logic with no meaningful test coverage where missing coverage represents a real risk.

   Flag a gap only if ALL of the following are true:
   - The section contains real logic (not boilerplate, getters, type definitions, logging,
     or framework wiring).
   - No existing test exercises this code path in a meaningful way.
   - A bug here could plausibly cause user-facing or data-integrity harm.

   Highest-value gaps to look for: core business logic with no test at all; error handling branches (upstream failures, malformed input, unavailable resources); boundary conditions (empty inputs, single-item collections, null/undefined propagation, type coercions); state mutations to persistent storage with no verification; integration seams where data is transformed between modules (serialization, timezone handling, currency rounding, pagination math).

   Do NOT flag: pure passthrough functions, framework lifecycle hooks that are impractical to unit test, trivial getters/setters, UI rendering details, code already covered by an existing test even if only partially, or code where the test contains an explicit comment saying why something was skipped.

   For each material gap, produce this block:

   ```
   SOURCE FILE: <relative path>
   LOCATION: <function or method name, plus line range if helpful>
   RISK: HIGH | MEDIUM | LOW
   WHAT IS UNTESTED: Plain-language description of the logic with no coverage.
   WHY IT MATTERS: One sentence on what could go wrong if this code has a bug.
   SOURCE CONTEXT: <function signature plus the 3-5 lines of logic most central to the gap, copied verbatim from the source>
   ACTION: A single imperative sentence telling the fixing LLM what test(s) to write. Specify the inputs, the expected output or behavior, and any edge case that must be covered. Start with a verb.
   ```

   After all gaps, output:

   ```
   PASS 2 SUMMARY
   Source files scanned: N
   Material gaps found: N
   All material gaps found:
     1. <file: function — one-line reason>
     2. <file: function — one-line reason>
     ...
   ```

5. **Write the combined report to `test-quality-audit.md` in the project's root test folder.**

   Use this exact structure:

   ```
   # Test Quality Audit
   _Generated: <today's date>_

   ## Part 1 — Misleading Tests

   ### <relative path to test file>

   > **TEST:** ...
   > **PATTERN:** ...
   > **SEVERITY:** 🔴 HIGH | 🟡 MEDIUM | 🟢 LOW
   > **FINDING:** ...
   > **ACTION:** ...

   (repeat for each finding; if no issues: _No issues found._)
   (repeat ### block for each file reviewed)

   ### Pass 1 Summary

   | Metric | Value |
   |--------|-------|
   | Files reviewed | N |
   | Tests reviewed | N |
   | Tests flagged | N |
   | PRIMARY flagged (A-D) | N |
   | SECONDARY flagged (E-H) | N |
   | High severity | N |
   | Medium severity | N |
   | Low severity | N |
   | Dominant pattern | ... |
   | Overall confidence | LOW / MEDIUM / HIGH |

   ---

   ## Part 2 — Coverage Gaps

   ### <relative path to source file>

   > **LOCATION:** ...
   > **RISK:** 🔴 HIGH | 🟡 MEDIUM | 🟢 LOW
   > **WHAT IS UNTESTED:** ...
   > **WHY IT MATTERS:** ...
   > **SOURCE CONTEXT:**
   > ```
   > <source snippet>
   > ```
   > **ACTION:** ...

   (repeat ### block for each source file with gaps)

   ### Pass 2 Summary

   | Metric | Value |
   |--------|-------|
   | Source files scanned | N |
   | Material gaps found | N |
   | High risk | N |
   | Medium risk | N |
   | Low risk | N |

   **High-risk gaps:**
   1. ...
   2. ...

   **Medium-risk gaps:**
   1. ...
   2. ...

   **Low-risk gaps:**
   1. ...
   2. ...
   ```

   Preserve all finding and action text exactly. No prose introductions, transitions, or commentary sections. No em dashes anywhere in the document.

   Confirm completion by telling the user: "Report written to test-quality-audit.md"
